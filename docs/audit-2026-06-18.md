# Loom Codebase Audit — Architect's Synthesis

## 1. Executive summary

Loom is structurally sound at the domain level (clean session lifecycle, well-isolated git/tmux packages, a deliberate unrecovered-cache for crash resilience) but carries one pervasive, systemic flaw: **the boundary between Bubble Tea's single-threaded Update loop and its concurrent `tea.Cmd` goroutines is not respected.** The same data race recurs in at least four subsystems — script dispatch, tmux capture, UI panes, and storage — and is independently confirmed by multiple agents. The second-largest theme is a cluster of **persistence/recovery correctness bugs** (orphan discovery, workspace migration, schema migration, title-derived identity) that can silently lose or misadopt a user's worktrees. A long tail of **fork-legacy debt** (stale comments, dead code, false doc invariants) actively misleads maintainers and, in several cases, masks the real bugs. Overall health: the core works, but the concurrency model needs a single disciplined fix applied uniformly, and the recovery paths need hardening before they cause irreversible data loss.

## 2. Cross-cutting themes

### Theme A — The `tea.Cmd`-goroutine data race (the dominant defect)
This is the single most important finding, reported six times against four different subsystems. Every returned `tea.Cmd` runs in its own goroutine concurrently with `Update`/`View`, yet code throughout the app mutates shared model state from inside those Cmds with no synchronization:
- **Script dispatch** mutates `m.list`/`m.splitPane`/`m.slots`/`m.focusedSlot` from the dispatch goroutine — "Script dispatch mutates the model ... from a Cmd goroutine" (`app/app_scripts.go`), "Script sync primitives mutate shared model state" (`app/app_scripts.go`), "Sync host primitives mutate model state read by View()" (`script/host.go`), and the xc-architecture restatement.
- **Tmux** races `ptmx`/`monitor` between the metadata fan-out and attach/kill lifecycle — "Data race on TmuxSession.ptmx and TmuxSession.monitor" and its xc-concurrency duplicate (`session/tmux/tmux.go`).
- **UI panes** mutate `PreviewPane`/`DiffPane`/`List` while `View()` reads them — "PreviewPane, DiffPane and List are mutated from the script Cmd goroutine" (`ui/preview.go`, `ui/list.go`, `ui/diff.go`).
- **Storage** races `unrecovered` and shared `*State` between concurrent pause/kill/save Cmds — "Data race on Storage.unrecovered and shared *State" (`session/storage.go`).

Why it matters: these are genuine data races under Go's memory model (undefined behavior), with realistic outcomes ranging from a torn `selectedIdx`, to corrupted `instances.json`, to nil-deref crashes when `loadSlot` swaps whole sub-model pointers mid-flight. They are invisible today only because CI builds `CGO_ENABLED=0` and so **cannot run `go test -race`** — the detector is structurally disabled.

### Theme B — False safety contracts that enable Theme A
Several comments and docs *assert* the very safety property the code lacks, which is why the races were written in the first place:
- The `TmuxSession` struct doc claims "documented accessors ... take the internal mutex" — there is no such mutex ("TmuxSession struct doc claims an 'internal mutex'", `session/tmux/tmux.go`).
- `host.go:60-64` and `app_scripts.go:131-137` assert the sync primitives "only mutate state that isn't also read by Update" — provably false.

Why it matters: a false invariant is worse than a missing one. Every one of these comments must be corrected *as part of* the race fix, or the next contributor re-introduces the bug.

### Theme C — Data-loss & misidentification windows in persistence/recovery
A coherent cluster of bugs in the orphan/migration/schema paths can silently lose or wrongly adopt user work:
- Orphan discovery only descends two levels, so multi-segment branches are "silently undiscoverable" — "Orphan discovery misses worktrees for branches with more than two path segments" (`session/orphan.go`).
- `workspace migrate` drops the per-user path segment with `filepath.Base`, hiding migrated worktrees from discovery, and does a raw rename without `git worktree repair` — "workspace migrate drops the per-user path segment" (`cmd/workspace_migrate.go`).
- A single forward-versioned record aborts the *entire* load in classic mode, or silently empties a whole workspace's instances on the multi-tab path — "Single forward-versioned or unknown record aborts the entire instance load" (`session/storage_migrate.go`).
- `workspace migrate` bypasses `session.Migrate` entirely, a latent corruption vector the moment a real migration step lands — "workspace migrate bypasses session.Migrate" (`cmd/workspace_migrate.go`).
- Lossy `HumanizeBranchLeaf` title derivation drives tmux identity, so recovery can kill a live agent session or spawn a zombie — "Orphan tmux liveness/restore keys on humanized title" and "Orphan recovery title derivation is lossy and unconditional" (`session/orphan.go`), feeding directly into "Orphan recovery can create a duplicate-title instance" (`app/app.go`).

Why it matters: these are the paths users hit *after* something already went wrong (a crash, an upgrade, a migration). Failure here turns a recoverable situation into permanent data loss, which is the worst possible time to be fragile. The asymmetry with the carefully-built per-record tolerance in `LoadAndReconcile` shows the resilient pattern already exists and simply wasn't applied uniformly.

### Theme D — `home` god-object and the dual-source-of-truth slot model
Three findings converge on `app/app.go`: the ~37-field `home` struct mixing storage/UI/overlay/workspace/orphan/script concerns ("home is a god object" ×2), and the structural root cause — the focused workspace slot has **two sources of truth** (`m.list` vs `m.slots[focusedSlot].list`) reconciled by manual `saveCurrentSlot`/`loadSlot` field-shuffling ("Focused workspace slot has two sources of truth", `app/app.go`).

Why it matters: this is not merely cosmetic. The "copy 5 fields out, copy 5 fields back" pattern is exactly what makes the Theme-A workspace-switch race so dangerous (it swaps whole sub-models) and what makes the startup-picker save-omission bug possible ("Startup workspace pick overwrites global list/storage without saving it first"). Every read path must special-case `i == m.focusedSlot`; forget it once and you read stale state. The slot mechanism being the *operational* swap (not incidental clutter) is the honest framing — the fix is to make the slot the single owner, not to cosmetically split the struct.

### Theme E — Fragile string-based identity and parsing
A recurring class of bug: behavioral decisions keyed on brittle string matching of CLI commands, git stderr, or magic names:
- Claude program matching uses two *different* predicates (`==` vs `HasSuffix`), neither matching the recovery command `"claude --continue"`, breaking auto-yes and trust-prompt dismissal after every crash recovery — "Recovery/flagged Claude sessions skip prompt detection" (`session/tmux/tmux.go`). **This is the highest-severity functional bug** — it silently disables a core advertised feature on exactly the restart path it exists to survive.
- `ApplyRecoveryFlag` corrupts the command on leading whitespace; the idempotency guard misses `--continue=VALUE` ("ApplyRecoveryFlag corrupts the command string", "Recovery-flag idempotency check misses", `session/agent/claude.go`).
- Branch-absence detection relies on English-only `strings.Contains(err, "not found")` ("Branch-absence detection relies on English-only substring matching", `session/git/worktree_ops.go`).
- Titles of only non-ASCII chars collapse to the bare prefix, colliding branches and risking deletion of a real branch ("Titles of only non-ASCII/special characters collapse to the bare branch prefix", `session/git/util.go`).
- `cs.sprintf` stringifies all args, breaking every numeric verb; adapter dispatch keys on the magic string `"default"` (`script/api.go`, `session/instance.go`).

Why it matters: these all stem from re-deriving identity from raw strings instead of threading structured data (the `agent.Adapter` already holds the authoritative prompt/trust config; locale is fixable with `LC_ALL=C`). Same root cause, many symptoms.

### Theme F — Swallowed errors and dead error-handling machinery
Errors are logged-and-discarded or handled by apparatus that never runs:
- A user's typed prompt is silently dropped on `SendPrompt` failure ("User-typed prompt silently dropped on SendPrompt failure", `app/app.go`).
- `rotatingWriter.Write` returns success after a failed reopen — total silent log loss ("rotatingWriter.Write swallows all write errors", `log/log.go`).
- The daemon's entire graceful-drain apparatus is dead because `StopDaemon` uses SIGKILL ("Daemon's signal handling and graceful drain are dead code", `daemon/daemon.go`).
- The deferred `Kill()` in `Start()` is a guaranteed no-op (`!started`), so its "releases resources" contract is false ("Start() error-cleanup defer calls Kill()", ×3, `session/instance.go`).

### Theme G — Fork-legacy debt (stale docs, dead code, wrong names)
A broad, low-severity tail that pollutes the codebase and *actively misleads*: the `tmux_unix.go`/`tmux_windows.go` platform split that doesn't exist (`session/tmux/doc.go`, CLAUDE.md); the phantom `RenameLegacySessions` call site (CLAUDE.md); `CleanupSessions` ignoring `claudesquad_` with a "session-" comment (`session/tmux/tmux.go`); the "claude squad binary" comment, dead `SanitizedToTmuxName`, dead `setPumpDest`, dead `MatchedIdx`, dead `UpdateInstance`, the discarded `PlaceOverlay` shadow, and the malformed `log.For("component", "daemon")` `!BADKEY`. Why it matters: individually trivial, collectively they erode trust in every comment and several of them (Theme B) mask real bugs.

## 3. Top architectural recommendations

Ordered by leverage (value ÷ effort). Items 1–2 are non-negotiable; the rest are high-value follow-ups.

**1. Make `go test -race` runnable, then route ALL script effects through `handleScriptDone` (kills Theme A at its largest source).** — *Blast radius: medium.* Add a CI target that builds with `CGO_ENABLED=1` solely to run the race detector. Then stop splitting script primitives into "sync" (mutate-now) vs "deferred" (intent); have **every** sync primitive (cursor/scroll/toggle/workspace-switch) only *record* a request on `scriptHost` under its existing `mu` and apply it on the main goroutine in `handleScriptDone`, exactly mirroring the `QueueInstance`/`AddInstance` pattern already present. **Bad historic decision called out:** the "sync primitives are safe because we're on the dispatch goroutine" model (and its false comments) was wrong from the start — the dispatch *is* the racing goroutine. *Risk of not doing it:* intermittent index-out-of-range panics, garbled selection, and crash-on-workspace-switch that will never reproduce in CI.

**2. Add a real mutex to `TmuxSession` guarding `ptmx`+`monitor` (kills Theme A's tmux source).** — *Blast radius: small/medium.* Add a `sync.Mutex`, snapshot `ptmx`/`monitor` into locals under the lock at the top of `CaptureAndProcess`/`HasUpdated`/`SendKeys*`/`Tap*`, and hold it across the swap/teardown in `Restore`/`Close`/`PausePreview` — **never** hold it across the blocking tmux subprocess I/O. Fix the struct doc that lies about an existing mutex. *Risk of not doing it:* nil-deref crash when an `alt+a` attach or kill overlaps a metadata tick on the selected instance.

**3. Make persistence concurrency-safe and migration-tolerant (Themes A + C).** — *Blast radius: medium.* Add a mutex to `*State` covering `InstancesData`/`HelpScreensSeen`/`lastWritten` (note `SetHelpScreensSeen` bypasses `Storage`) plus a mutex on `Storage.unrecovered`. Separately, make `MigrateAll` **skip-and-collect** per-record failures into a parallel raw-`json.RawMessage` cache so one forward-versioned record never aborts a whole load — apply the resilient pattern `LoadAndReconcile` already uses. *Risk of not doing it:* corrupted `instances.json`, or a single bad record from a version downgrade wiping an entire workspace's visible instances.

**4. Thread the `agent.Adapter` into `TmuxSession` instead of re-deriving program identity from raw strings (kills the worst of Theme E).** — *Blast radius: small/medium.* One source of truth for trust-prompt and update detection, replacing the two divergent `==`/`HasSuffix` predicates. This directly fixes the high-severity "Recovery/flagged Claude sessions skip prompt detection" bug and structurally prevents the flagged-profile variant. *Risk of not doing it:* auto-yes and trust-prompt auto-dismissal stay silently broken after every crash recovery and for every Claude profile carrying flags — a core feature failing exactly when it's needed.

**5. Harden the orphan/migration recovery paths (Theme C).** — *Blast radius: medium.* Walk `worktrees/` recursively and reconstruct branches from the full relative path (fixes deep-branch discovery and the `workspace migrate` `filepath.Base` truncation); match orphan tmux sessions by **worktree working directory** rather than `sanitize(humanize(branch))`; dedup recovered titles against the live list and selected orphans; add `git worktree repair` to the migrate move. *Risk of not doing it:* stranded unrecoverable worktrees, killed live agent sessions during startup cleanup, and corrupted by-title dedup invariants.

**6. Make the workspace slot the single source of truth (resolves Theme D's root cause).** — *Blast radius: large but mechanical.* Store `slots []*workspaceSlot`, hold only `focusedSlot`, expose a `focused()` accessor, and replace `m.list`/`m.storage`/`m.splitPane` accesses with `m.focused().*`. This deletes `saveCurrentSlot`/`loadSlot`'s field-shuffling, collapses every `i == m.focusedSlot ? ... : ...` branch, and structurally prevents the stale-slot and missing-save bugs. **Bad historic decision called out:** flattening focused-slot state onto `home` was the operational mechanism, but it made the slot-swap the most dangerous racing write and spread the focused-index special-case across the file. Best sequenced *after* item 1 so the swap is already off the racing goroutine. *Risk of not doing it:* every future feature touching slots risks reading stale state; the highest-churn file stays the hardest to reason about.

**7. Make `Start()` failure cleanup self-contained, not dependent on a no-op `Kill()` (Theme F).** — *Blast radius: small.* Have the deferred handler tear down the snapshotted `gw`/`ts` directly (or gate `Kill` on `tmuxSession!=nil || gitWorktree!=nil` rather than `started`), and make `Setup` self-cleaning on any error after `worktree add`. Fix the false comment. *Risk of not doing it:* the next fallible step added between `Setup` and the inline cleanup silently leaks a worktree + branch + tmux session.

# Appendix — All confirmed findings


## HIGH severity


### 1. Script dispatch mutates the model (list/splitPane/slots) from a Cmd goroutine, racing with Update
`concurrency` · `app/app_scripts.go:131-176, 342-378` · _unit: app-core_

**Problem.** dispatchScript returns a tea.Cmd whose body calls m.scripts.Dispatch(ctx, key, host). Bubble Tea runs every Cmd in its own goroutine concurrently with the Update loop (confirmed in vendored tea.go:346-354 — `go func(){ msg := cmd(); p.Send(msg) }()`). Inside Dispatch, the Lua handlers invoke scriptHost sync primitives — CursorUp/CursorDown (m.list.Up/Down), ToggleDiff (m.splitPane.ToggleDiff), ListPageUp/Down/Top/Bottom, ScrollLineUp..., and crucially WorkspacePrev/WorkspaceNext which call m.saveCurrentSlot()+m.loadSlot()+m.updateTabBarStatuses(). These mutate m.list, m.splitPane, m.slots, and m.focusedSlot. ui.List and ui.SplitPane have no internal locking (ui/list.go:101 — plain fields items/selectedIdx/scrollOffset). Meanwhile the main Update goroutine reads/writes the same fields: previewTickMsg reads m.list.GetSelectedInstance() (app.go:631), tickUpdateMetadataMessage iterates slot.list.GetInstances() and m.focusedSlot (app.go:666-676), updateTabBarStatuses reads slot.list (app.go:1988). The comment at app_scripts.go:131-137 asserts 'we are still on the dispatch goroutine ... Update has not had a chance to race with us' — this is false; the dispatch IS the goroutine. Result: data race on selectedIdx/items/focusedSlot, e.g. a workspace-switch script firing while a metadata tick reads the slot list, or CursorUp running while previewTick reads the selection. `go test -race` or a fuzzed keymap would surface it; in production it manifests as occasional index-out-of-range panics or a stale/garbled selection.

**Fix.** Do not mutate model state from inside the Dispatch Cmd. Either (a) run the engine dispatch but have sync primitives record intents that are applied on the main goroutine in handleScriptDone (same pattern QueueInstance already uses for AddInstance), or (b) marshal all navigation/diff/workspace mutations into the returned scriptDoneMsg and perform them in Update. At minimum, fix the misleading comment and add a `-race` test that fires a workspace-switch script concurrently with a metadata tick.

**Verifier note.** The recommendation is sound and complete; no correction needed. Preferred fix is option (a): have the sync primitives record intents (or buffered navigation deltas) on the scriptHost under its existing mu, then apply them on the main goroutine in handleScriptDone — exactly mirroring the QueueInstance/AddInstance pattern already present at app_scripts.go:98-109 and 527-531, which the codebase adopted for precisely this reason. Additionally: (1) the false comment at app_scripts.go:131-137 and the incorrect safety claim in the Host doc at host.go:60-64 must both be corrected, since they actively mislead future maintainers into thinking the path is safe; (2) a `go test -race` test firing a default-keymap navigation/workspace-switch dispatch concurrently with a metadata/preview tick would surface it. One nuance worth flagging to the maintainer: ToggleDiff/scroll primitives mutate splitPane viewport state that is ALSO touched on the Update goroutine (e.g. previewTickMsg calls m.splitPane.UpdateTerminal at app.go:645), so the splitPane mutations are equally racy, not just list/slots — the fix must cover all sync primitives, not only the navigation/workspace subset.


### 2. Script sync primitives mutate shared model state off the main goroutine, racing the preview/metadata tick loops
`concurrency` · `app/app_scripts.go:131-176, 184-238, 342-378` · _unit: app-state_

**Problem.** dispatchScript returns a tea.Cmd that runs m.scripts.Dispatch in a Bubble Tea Cmd goroutine (app_scripts.go:359-360). Inside that Dispatch, cs.actions.* synchronous primitives call straight into the scriptHost, which mutates shared *home / *List / *SplitPane / slot state on the dispatch goroutine: CursorUp -> s.m.list.Up() (line 141), WorkspacePrev/Next -> s.m.saveCurrentSlot()/loadSlot()/updateTabBarStatuses()/persistFocusedWorkspace() (lines 155-176, which swap m.list, m.splitPane, m.slots, m.focusedSlot, m.activeCtx, m.tabBar wholesale per app.go:1573-1611), ToggleDiff and all Scroll*/List* pass-throughs. Meanwhile the main Update goroutine is NOT quiescent: the self-rearming previewTickMsg (every 33-100ms, app.go:606-659) reads m.list.GetSelectedInstance() and mutates m.splitPane via instanceChanged(), and tickUpdateMetadataMessage (app.go:663-685) iterates m.slots/m.focusedSlot/m.list. ui.List has no internal lock (Up() reads/writes l.items and l.selectedIdx, list.go:554-565). So a key press that triggers a script while a tick is delivered is a genuine unsynchronized concurrent read/write on m.list, m.splitPane, m.slots, m.focusedSlot, m.activeCtx — the Go race detector would flag it. The comment at app_scripts.go:131-137 ('the engine holds its mutex during dispatch and we are still on the dispatch goroutine ... Update has not had a chance to race with us') is incorrect: the engine mutex serializes script-vs-script, not script-vs-Update, and the tick loops guarantee Update IS running concurrently.

**Fix.** Do not mutate model/list/splitPane/slot state from inside the engine dispatch goroutine. Route all sync primitives the same way as Intents: have the host record the requested mutation (an enum or closure) and apply it in handleScriptDone on the main goroutine, OR run the entire dispatch synchronously on the Update goroutine (the engine mutex already serializes, and Lua handlers are short). Verify with `go test -race ./app/...` driving a script bind that calls cursor_up while the preview tick fires.

**Verifier note.** Severity is more defensibly "high" than "critical". The race is real but conditional: it only fires for keys bound to sync primitives (cursor_up/down, toggle_diff, workspace_prev/next, scroll/list pass-throughs), and only when a tick message lands in the same narrow window as the keypress-triggered Cmd goroutine. The most common outcome is a corrupted selectedIdx or a stale/torn read rather than guaranteed corruption every press. That said, the workspace-switch path (loadSlot reassigning m.list/m.splitPane pointers mid-flight) can plausibly cause a nil-deref/crash, which keeps it well above medium. The recommendation is correct and complete: route sync primitives through handleScriptDone on the main goroutine the same way Intents are (record a closure/enum on the host, apply in handleScriptDone), OR run the whole dispatch synchronously on the Update goroutine since the engine mutex already serializes and Lua handlers are short. The verification note is also right that a new -race test exercising the tick concurrently is needed — the current synchronous cmd() test invocation would not surface it.


### 3. Sync host primitives mutate model state read by View(), racing the Bubble Tea main loop
`concurrency` · `script/host.go:60-71` · _unit: script-engine_

**Problem.** host.go documents the sync primitives (CursorUp/CursorDown/ToggleDiff/WorkspacePrev/WorkspaceNext and the scroll/list jumps) as safe to call on the dispatch goroutine because they 'only mutate state that isn't also read by Update at the same time.' That contract is violated by the workspace-switch primitives. Engine.Dispatch runs inside a tea.Cmd goroutine (app/app_scripts.go:359-360), concurrently with View()/Update() on the main goroutine. WorkspaceNext/WorkspacePrev (app/app_scripts.go:155-176) call m.loadSlot, which reassigns m.list (app/app.go:1593), m.splitPane (1594) and m.focusedSlot (1591) — the very pointers View() and Update() dereference every frame. Swapping these struct pointers from a background goroutine while the renderer reads them is a data race on the home model, not merely 'theoretical.' CursorUp/ToggleDiff/scroll also mutate ui widget internals concurrently with rendering.

**Fix.** Route workspace switching (and ideally all model-mutating primitives) through the Enqueue+Intent path so the mutation lands in Update on the main goroutine, or guard the affected home fields with a mutex. Update the host.go contract comment to stop claiming these are race-free. Run the package under `go test -race` exercising a script that calls cs.actions.workspace_next while the UI renders.

**Verifier note.** One technical imprecision in the finding's mechanism, which does not change the verdict: in bubbletea v1.3.4 `View()` is NOT called concurrently by the renderer goroutine. `Update(msg)` and `View()` are both invoked inline on the single eventLoop goroutine (tea.go:508-510), and the standard renderer's async `flush()` reads only a pre-computed string buffer, not the model. So the concurrent reader of `m.list`/`m.splitPane`/`m.focusedSlot`/`m.slots` is the eventLoop goroutine running `Update` for an interleaved recurring tick (previewTickMsg / tickUpdateMetadataMessage / spinner) and the View that follows it — not the renderer. The race is still real and the cited write sites are correct; only the identity of the concurrent reader should be restated as "the eventLoop goroutine processing an interleaved ticker message" rather than "the renderer reading them." The recommendation is correct and complete: route workspace switching through Enqueue+Intent so the mutation lands in Update on the eventLoop goroutine, or guard these home fields with a mutex, and fix the host.go / app_scripts.go contract comments. CursorUp/CursorDown/ToggleDiff/scroll/list-jump primitives also mutate ui widget internals (m.list, m.splitPane) on the dispatch goroutine concurrently with the same tick-driven Update/View reads, so they share the same defect class, though the workspace-switch pointer swaps are the most dangerous (they replace whole sub-models rather than mutate internal fields).


### 4. Data race on Storage.unrecovered and shared *State from concurrent tea.Cmd goroutines
`concurrency` · `session/storage.go:103-124 (SaveInstances), 182-207 (DeleteInstance), 283-295 (dropUnrecovered)` · _unit: session-persistence_

**Problem.** Storage has no mutex (no sync.* field), yet its mutable state — the `unrecovered` slice and the wrapped *config.State (InstancesData, lastWritten) — is mutated from inside tea.Cmd closures that Bubble Tea runs in separate goroutines. pauseActionFor/resumeCmd call m.storage.SaveInstances (intents.go:312, 334) and killAction calls m.storage.DeleteInstance → dropUnrecovered (intents.go:226), all from returned Cmd closures, not from Update. Bubble Tea executes each returned Cmd in its own goroutine, so two such Cmds (e.g. a pause-save for one instance and a kill-delete for another, or a save racing the orphan-recovery save) can run concurrently and read/write `s.unrecovered` and `state.InstancesData`/`state.lastWritten` with no synchronization. This is a textbook data race that `go test -race` would flag and can corrupt the in-memory cache or the bytes written to instances.json.

**Fix.** Add a sync.Mutex to Storage guarding unrecovered + all state mutations, and lock it in SaveInstances/LoadAndReconcile/DeleteInstance/UpdateInstance/DeleteAllInstances/dropUnrecovered/UnrecoveredWorktreePaths. Alternatively, funnel all persistence through the Update goroutine (return a message instead of saving inside the Cmd). At minimum add a `go test -race` run over the app package to surface this.

**Verifier note.** The recommendation is essentially correct (add a sync.Mutex guarding unrecovered + state mutations, or funnel saves through a returned message handled in Update). One refinement: a mutex placed only on *Storage does cover the instances.json path, because every instances save in the app goes through Storage.SaveInstances/DeleteInstance/UpdateInstance/DeleteAllInstances (including the Update-goroutine save at app.go:952), and Storage internally calls state.SaveInstances. ### 12. Registry Add/Remove/Rename clobber concurrent writers (no reload-before-save)
`concurrency` · `config/workspace.go:125-156, 159-180, 218-248` · _unit: config_

**Problem.** UpdateLastUsed (258-274) and SetOpenWorkspaces (290-309) were deliberately hardened to reload the on-disk registry before saving so a second process is not clobbered by a stale in-memory snapshot — see their doc comments ('reloaded before save so concurrent writers ... are not clobbered'). But Add, Remove, and Rename do NOT do this: each mutates the receiver's snapshot and then calls SaveWorkspaceRegistry(r) directly, writing the whole file. Two concurrent `loom workspace add foo` / `loom workspace add bar` (each does LoadWorkspaceRegistry then Add) both load the same base set, append their own entry, and write the full snapshot — last writer wins and the other workspace is silently lost. Worse, a running TUI calling Remove/Rename will overwrite (drop) any workspace a concurrent `loom workspace add` in another shell just persisted, because the TUI's long-lived registry never reloaded. The CLAUDE.md and method docs explicitly anticipate 'a `loom workspace add` invocation in another shell, or a second running TUI', so this is a known concurrency model that Add/Remove/Rename violate.

**Fix.** Make Add/Remove/Rename use the same load-merge-save discipline as UpdateLastUsed/SetOpenWorkspaces: reload the registry from disk inside the method, apply the mutation to the fresh copy, save, then sync the receiver's in-memory fields. Alternatively serialize all registry writes through a file lock (e.g. an O_EXCL lock file in the global dir) so add/remove/rename across processes cannot interleave.

**Verifier note.** The bug is real but I'd rate it medium rather than high. This is a single-user, local, interactive CLI/TUI; registry mutations (add/remove/rename) are infrequent, manual operations, so true concurrent writes are an uncommon scenario. Impact is silent data loss of a registry entry, which is recoverable (re-run `workspace add`), and no integrity corruption occurs because AtomicWriteFile still produces a well-formed file. That said, the underlying invariant is documented and deliberately enforced elsewhere, so it is a legitimate defect.

The recommendation is correct but needs a caveat: a naive reload-merge for Add must re-run the uniqueness checks (name AND path) against the freshly loaded snapshot before appending, not merely append, or it can resurrect/duplicate entries; Remove and Rename must tolerate the target having been concurrently removed/renamed on disk (currently they'd error 'not found' or silently no-op against the fresh copy). Because of these merge subtleties across three different mutation shapes, the file-lock alternative (an O_EXCL/flock lock in the global dir wrapping load+mutate+save) is the cleaner and safer fix and should be preferred.

Its only caller is the `loom reset` subcommand (main.go:214), and that path does NOT run RenameLegacySessions first — it deletes all instances (main.go:209) then calls CleanupSessions. So a user who upgrades from claude-squad with live legacy tmux sessions and runs `loom reset` will have their loom_ sessions cleaned but their claudesquad_ sessions left orphaned, contradicting CLAUDE.md's 'orphan sweep accepts both prefixes' contract. Separately, the doc comment 'kills all tmux sessions that start with "session-"' is plainly false — there is no 'session-' prefix anywhere in the code.

**Fix.** Either include LegacyTmuxPrefix in the CleanupSessions match (mirror the reconcile.go OR-prefix check) so `loom reset` sweeps legacy sessions too, or call RenameLegacySessions before CleanupSessions in the reset path. Fix the doc comment to describe the loom_ prefix it actually matches.

**Verifier note.** Severity is overstated at medium. The CLAUDE.md "orphan sweep accepts both prefixes" contract refers specifically to the reconcile orphan sweep (CleanupOrphanedSessions), which does honor both prefixes; CleanupSessions is a separate `loom reset`-only path, so this is not a direct violation of that named contract. Impact is limited to a one-time migration window with live legacy sessions, causes no data loss or correctness break, and the leftover claudesquad_ sessions are harmless stragglers that the normal-launch reconcile orphan sweep self-heals on the next `loom` run. Combined with a false-but-cosmetic doc comment, low is the right rating. The recommendation itself is correct and complete: either OR LegacyTmuxPrefix into the CleanupSessions regex/match (mirroring reconcile.go:208) or call RenameLegacySessions before CleanupSessions in the reset path, and fix the "session-" doc comment to say loom_.


### 76. workspace migrate bypasses session.Migrate, so schema versioning is unenforced on the migration path
`bug` · `cmd/workspace_migrate.go:86-115` · _unit: xc-persistence-flow_

**Problem.** planMigration json.Unmarshal's global state directly into []migrationInstance and re-marshals via mergeWorkspaceInstances, never routing records through session.Migrate / MigrateAll. The CLAUDE.md 'gotcha' states all records pass through Migrate so callers receive CurrentSchemaVersion data and that future/unsupported versions are rejected (session/storage_migrate.go:29-31 returns an error for SchemaVersion > CurrentSchemaVersion). The migrate CLI honors neither: a record written by a NEWER binary (schema_version=2) is silently copied into a workspace state.json by an older binary running `workspace migrate`, and a v0 record is moved without its v0->vN field upgrades being applied. Today v0->v1 is a no-op so this is latent, but the moment a real migration step is added, `workspace migrate` becomes a silent corruption vector: it transplants un-upgraded or down-rev records into workspace stores. The drift-guard test (cmd/workspace_migrate_shape_test.go) checks field shape/kind but cannot catch this — it never exercises Migrate.

**Fix.** Route the global InstancesData through session.MigrateAll before planning, and re-marshal the upgraded InstanceData (or at minimum reject records whose schema_version exceeds the binary's CurrentSchemaVersion) so the migrate CLI cannot transplant un-upgraded or future-version records.

**Verifier note.** Severity: medium is defensible but slightly generous. At the current single non-trivial schema version the impact is strictly zero (v0 and v1 payloads are byte-identical and the only reject case — future schema_version — cannot exist when no newer binary has shipped). It is a latent/forward-looking defect, so low-to-medium is more accurate than medium.

Recommendation caveat: the auditor's primary suggestion to "route the global InstancesData through session.MigrateAll" conflicts with a documented architectural constraint. cmd/workspace_migrate.go:21-26 deliberately duplicates the InstanceData shape as migrationInstance specifically to avoid a session->cmd import cycle (cmd is already imported by session via cmd.Executor). Importing session.MigrateAll into cmd would reintroduce that cycle. The faithful fix is the auditor's own fallback: mirror CurrentSchemaVersion as a constant in cmd and reject (or locally upgrade) records whose schema_version exceeds it inside planMigration, and extend cmd/workspace_migrate_shape_test.go to cover the version guard. The recommendation should lead with the fallback, not the MigrateAll route.


### 77. Orphan recovery title derivation is lossy and unconditional, risking re-adoption under a wrong identity
`bug` · `session/orphan.go:208-218` · _unit: xc-persistence-flow_

**Problem.** When state.json is rewritten without removing a worktree (the exact scenario the recent data-loss fixes target), the original instance Title is unrecoverable, so DiscoverOrphans reconstructs it with HumanizeBranchLeaf — last '/'-segment with dashes turned to spaces. This is irreversible and collides easily: branches 'aidanb/fix-bug' and 'bob/fix-bug' both humanize to title 'fix bug'. Recovering both (or recovering one whose humanized title matches a live instance) feeds directly into the duplicate-title problem above. Because the recovered InstanceData also sets SessionName: cand.Title, the tmux session it adopts/creates is keyed on this lossy title too, so a mismatch against the originally-running loom_<oldtitle> session means HasLiveTmux may be false even when a live pane exists under the real prior title, and recovery spawns a fresh PTY while the old one leaks.

**Fix.** Persist enough to recover the original title (e.g. write a small sidecar in each worktree dir at creation, or read the branch's first commit subject) so recovery isn't title-lossy; failing that, dedup recovered titles against the live list and existing tmux sessions before adoption.

**Verifier note.** The recommendation is sound but should note the dominant symptom: because the same lossy title also defeats the CleanupOrphanedSessions exemption (claimedTitles uses cand.Title at app.go:363-365 while the real session is keyed on the original Title, and ToLoomTmuxName is case-sensitive), the more likely failure is that the user's still-running agent session is KILLED during startup cleanup, not leaked. The PTY-leak symptom is reachable mainly on the multi-tab restore path where CleanupOrphanedSessions is skipped. Both fixes proposed (persist original title via worktree sidecar at creation, and/or dedup recovered titles against the live list and existing tmux sessions) are appropriate; additionally, the tmux probe/exemption should be keyed off a stable identity rather than a lossy reconstructed title.
